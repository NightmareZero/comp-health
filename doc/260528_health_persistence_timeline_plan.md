# 260528_health_persistence_timeline_plan.md

## 背景

当前 `comp-health` 已具备基础的 `server` / `agent` 双模式能力，但状态存储仍为内存实现：

- 进程重启后历史与当前状态都会丢失
- 时间线点位数量固定且较短
- `retention` 配置尚未真正驱动历史裁剪
- 时间线悬浮信息不足，无法展示采样时间、耗时、说明、节点来源

用户当前约束：

- 最多监听 **20 个服务**
- 保留单二进制部署风格
- 优先考虑 **高性能、低复杂度、可恢复** 的实现
- 时间范围希望支持切换：`6h / 12h / 3d / 7d`
- 不同时间范围下点位密度应尽量接近

## 目标

实现一套适合小规模高可靠场景的状态持久化与历史展示方案：

1. 重启后可恢复当前状态与历史样本
2. 可配置历史保留时长
3. 前端支持切换不同时间窗口
4. 后端按固定目标点数聚合，避免前端渲染过量原始点
5. 时间线支持悬浮查看采样时间、耗时、说明、节点信息

## 选型结论

在 **最多 20 个服务** 的规模下，采用：

- **纯 Go SQLite 驱动**
- **原始采样持久化**
- **最新状态表 + 历史结果表**
- **后端固定密度聚合**
- **前端只渲染聚合点**

这是性能、部署复杂度和可维护性之间最平衡的方案。

## 容量估算

按最常见配置估算：

- 服务数：20
- 采样频率：每 30 秒 1 次
- 单服务每天采样数：2880
- 全部服务每天采样数：`20 × 2880 = 57600`
- 7 天约：40 万级样本

该数量级对 SQLite 完全可接受，性能瓶颈主要在：

- 索引设计
- 查询方式
- 是否返回全量原始点
- 是否存在每次写入同步清理历史的低效路径

## 实施方案

### 1. 数据模型扩展

扩展 `StatusPoint`，保留时间线 tooltip 所需信息：

- `at`
- `status`
- `latency_ms`
- `message`
- `node_id`
- `node_name`

必要时增加聚合辅助字段，例如：

- `sample_count`
- `bucket_start`
- `bucket_end`

但第一版可以先保持简洁，只返回前端需要展示的字段。

### 2. Store 接口升级

当前 `Store` 只有：

- `SaveResult`
- `SaveReport`
- `Snapshot`
- `Nodes`

需要扩展为支持：

- 查询指定时间窗口的历史点
- 由后端聚合历史时间线
- 触发过期数据清理
- 启动后直接恢复最新状态

### 3. SQLite 存储设计

建议新增：

- `internal/store/sqlite.go`

第一版可使用两张表：

#### `check_results`
保存历史原始采样点：

- `id`
- `node_id`
- `node_name`
- `probe_id`
- `name`
- `type`
- `target`
- `status`
- `latency_ms`
- `message`
- `checked_at`

建议索引：

- `(probe_id, checked_at)`
- `(node_id, probe_id, checked_at)`
- `(checked_at)`

#### `latest_status`
保存每个 `node_id + probe_id` 的最后状态：

- `node_id`
- `probe_id`
- 其余展示所需字段
- `checked_at`

主键建议：

- `(node_id, probe_id)`

这样启动恢复首页时，不需要扫描历史表。

### 4. 写入路径优化

- 本地 probe 结果：单条写入即可
- agent 上报结果：整批结果放进一个事务
- 同一事务中：
  - 插入 `check_results`
  - upsert `latest_status`

这样可以减少磁盘同步次数，并保持状态一致。

### 5. 保留策略

将现有 `server.retention` 真正用于历史清理。

建议策略：

- 仅清理 `check_results` 中过期数据
- `latest_status` 永远保留最新一条
- 清理任务后台周期性执行，例如每 1 小时或每 6 小时一次
- 不要在每次写入时顺带清理

### 6. 历史 API 设计

现有 `/api/v1/status` 需要支持时间范围参数，例如：

- `range=6h`
- `range=12h`
- `range=3d`
- `range=7d`

同时可内部设定目标点数，例如：

- 默认 `96` 点

后端流程：

1. 根据时间范围计算窗口
2. 查询该窗口内原始点
3. 按目标点数切分时间桶
4. 每个桶输出一个代表点

### 7. 聚合规则

推荐规则：

- 每个桶优先取 **最差状态**
- 代表时间取桶内最近采样时间
- 延迟可取最近值或最大值
- message 优先保留异常信息
- node 信息沿用样本来源

这样能避免故障被大量正常点平均掉。

### 8. 前端展示改造

需要修改：

- `internal/webfs/web/index.html`
- `internal/webfs/web/app.js`
- `internal/webfs/web/styles.css`

改造目标：

- 增加时间范围切换 UI：`6h / 12h / 3d / 7d`
- 时间线消费后端聚合点
- 鼠标悬浮时显示：
  - 采样时间
  - 耗时
  - 结果说明
  - 节点来源
- 时间线点数不随范围线性膨胀

### 9. 兼容性策略

- 保留 `memory` 驱动，作为开发或回退方案
- 默认可以继续兼容旧配置
- 若配置 `storage.driver: sqlite` 才启用持久化
- 前端对缺失字段做容错

## 第一阶段实施范围

本轮开始实施，建议先完成：

1. 扩展模型字段
2. 扩展 `Store` 接口
3. 新增 SQLite store
4. 配置结构支持 SQLite
5. server 端支持按配置选择 store
6. `/api/v1/status` 支持 `range` 查询与固定密度聚合
7. 前端支持范围切换与 tooltip

## 暂不纳入第一阶段

以下内容建议先不做：

- 复杂图表库
- 明细页钻取
- 多实例共享数据库
- WebSocket 实时推送
- 外部归档存储

## 验证重点

1. 使用 SQLite 启动后，写入状态并重启，确认状态恢复成功
2. 验证 `retention` 会清理历史，但不影响最新状态恢复
3. 在 `6h / 12h / 3d / 7d` 下，点数保持接近固定值
4. tooltip 可正常展示时间、耗时、说明、节点
5. 20 个服务规模下，接口响应和页面渲染保持流畅

## 结论

对于当前项目规模，最高性价比的方案不是更换更重的数据库，而是：

- 用 SQLite 做持久化
- 用 `latest_status` 加速启动恢复
- 用后端聚合控制历史查询成本
- 用固定密度时间线保证前端稳定展示

该方案实现复杂度适中，且后续仍可自然扩展到服务详情页、明细查询和更长历史范围。
